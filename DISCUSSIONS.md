# Discussion Strategy

## Purpose

GitHub Discussions should host open-ended community conversations. Use issues for actionable, bounded work and private reporting for vulnerabilities.

## Recommended Categories

| Category      | Purpose                                    | Example                                              |
| ------------- | ------------------------------------------ | ---------------------------------------------------- |
| Announcements | Maintainer updates and releases            | “v0.2 milestone scope announced”                     |
| Architecture  | Design choices and ADR proposals           | “Should this decision require an ADR?”               |
| Dispatcher    | Dispatcher learning and structure ideas    | “How should Dispatcher examples be reviewed?”        |
| Performance   | Measurement and performance topics         | “What evidence should performance claims include?”   |
| Interview     | Interview-handbook structure and review    | “How should question difficulty be classified?”      |
| Java          | Enterprise Java scope and sources          | “Which specifications are authoritative?”            |
| AEM           | General AEM handbook topics                | “Where should this AEM concept live?”                |
| General       | Introductions and cross-cutting discussion | “How are you using the learning path?”               |
| Ideas         | Early proposals not ready for an issue     | “Could the handbook support role-based collections?” |

## Discussion Template

A useful opening post states the context, desired outcome, evidence or alternatives, and the specific community input requested. Do not post credentials, proprietary examples, customer information, or embargoed security details.

## Moderation and Outcomes

Maintainers may recategorize, close, or convert discussions into issues. Decisions with lasting architectural impact should be captured in an ADR; accepted delivery work should be tracked on the project board.
